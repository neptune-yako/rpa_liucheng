# 省系统扫码登录中间件-核心业务理解与流程拆解

> - **本篇重点**：聚焦于理解中间件在“客户端-服务端”通信中的桥梁作用，分析其轮询分配端口的底层逻辑与常驻服务流程。

---

## 1. 为什么需要中间件？（设计意图与架构定位）

在整个扫码登录体系中，**中间件**扮演着“**通信调度员**”的角色：

* **多实例并发分流**：为了支持多个账号的并发处理，服务器上可能会同时启动多个 **RPA 服务端实例**（例如一个监听 `13002` 端口，另一个监听 `13003` 端口）。
* **屏蔽连接细节**：**RPA 客户端**在发起请求时，并不知道当前哪一个服务端实例是空闲的，也不应该去硬编码连接某个具体的端口。
* **统一接入点**：客户端统一连接本地回路的固定端口 **`127.0.0.1:13001`**（即中间件）。中间件采用 **轮询（Round-Robin）算法**，将接入的客户端请求依次分配给配置好的各个服务端端口，从而实现服务端的负载均衡与分流。

---

## 2. 文字描述的中间件 RPA 流程执行过程

中间件在后台以独占套接字的方式常驻运行，其具体的运行与分发逻辑如下：

1. **环境准备与初始化**：流程启动后，首先初始化 Socket 对象（`sock = socket.socket()`）。
2. **绑定监听端口**：将 Socket 绑定到本地回路 IP 的固定端口 `127.0.0.1` 和 **`13001`** 上。
3. **启动监听队列**：调用 `sock.listen(5)` 开始侦听，最大允许 5 个连接请求在队列中排队。
4. **解析可用端口池**：读取流程输入参数 `服务端端口列表`（由逗号分隔的端口字符串，如 `"13002, 13003"`），替换中文逗号和空格，并切分成端口列表数组。
5. **进入常驻外层循环**：开启 `while True` 无限死循环，确保服务不间断运行。
6. **循环轮询端口数组**：通过 `for 端口 in 端口列表` 遍历可用端口池。当遍历到列表尾部时，会自动回到列表首位重新循环，实现经典的轮询分配。
7. **阻塞挂起并等待连接**：在当前端口指针下，调用 `conn, address = sock.accept()`。流程在此处阻塞挂起，直到有客户端连接到 `13001` 端口。
8. **接收客户端握手消息**：客户端接入后，读取其发送的最多 4096 字节的握手请求（通常为 `"请分配一个端口"`），并在控制台解码打印。
9. **发送分配端口给客户端**：将当前轮询指向的 `端口` 转换成 UTF-8 字节流，调用 `conn.send` 发送给连接的客户端。
10. **记录日志并释放连接**：通过内部 `logger` 记录本次分配的端口号及成功发送记录，随后关闭本次 TCP 物理连接，解除阻塞，指针移动到下一个端口，回到第 7 步继续挂起等待。

## 3. 运营与常见异常故障排查

### Q1: 中间件启动失败，提示端口被占用如何处理？
* **现象**：启动流程报错，提示 `bind` 失败，Address already in use。
* **原因**：本地 `13001` 端口已被其他进程（如未完全关闭的旧中间件实例，或其他本地服务）占用。
* **排查与解决**：
  1. 打开命令行执行：`netstat -ano | findstr 13001`。
  2. 找到占用该端口的进程 PID。
  3. 执行 `taskkill /F /PID {PID}` 强行终止占用进程，随后重新启动中间件。

### Q2: 客户端连接超时，无法获取分配端口？
* **排查顺序**：
  1. **检查中间件运行状态**：确认中间件 RPA 进程正处于“正在运行”状态，并未停止或崩溃。
  2. **检查端口防火墙**：虽然连接的是 `127.0.0.1` 本地回路，但仍需确认本地安全软件没有拦截中间件的 Socket 侦听操作。
  3. **核对端口列表格式**：确保流程输入的 `服务端端口列表` 格式正确，用逗号分隔，且列表中填写的服务端端口（如 `13002`）在后台确实有对应的服务端机器人正在进行接收监听。
