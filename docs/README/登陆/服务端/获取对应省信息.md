# 获取对应省信息子流程-核心业务逻辑与流程拆解

> - **本篇重点**：聚焦于服务端核心子流程 `获取对应省信息` 的执行细节，分析登录成功后 Cookie 凭证的拆解提取、后台个人中心跳转以及实名信息爬取的业务闭环。

---

## 1. 为什么需要这个子流程？（设计意图与架构定位）

本流程作为用户扫码成功后的“**收尾与凭证收割节点**”。一旦主流程判定用户已成功扫码并跳转进入省系统首页，便会立即调度本流程：

* **Cookie 凭证拆解**：直接抓取 Chrome 浏览器中最新写入的 Cookie 键值对大文本，并通过高精度的分割算法，剥离提取出后续同步数据必不可少的 `SESSION` 和 `openstack_cookie_insert` 会话值。
* **物理账号解密映射**：仅凭 Cookie 无法与本地的用户账号体系进行精确绑定。机器人需要鼠标悬停并点击进入后台的“账号设置”个人中心，将当前扫码用户的真实“省账号”和“组织机构”爬取下来，供数据库回填映射。

---

## 2. 文字描述的流程执行过程

整个子流程在执行时采用了双重尝试与异常捕获防护。具体的流转与提取过程如下：

### 第一阶段：凭证获取与格式化分割
1. **双重重试与异常保护**：开启一个最大次数为 2 次的 For 尝试循环，并将核心抓取逻辑完全包裹在 `Try-Catch` 异常保护块中。如果在第一次执行时遭遇网络波动或页面假死，会自动捕获错误并触发第二次重试机会。
2. **抓取浏览器 Cookie**：通过网页连接对象抓取 Chrome 浏览器中当前的完整 Cookie 字符串。
3. **高精度字符串分割**：
   * **提取 Cookie 键值对**：将 Cookie 字符串按照分号（`;`）进行第一层切分，得到各个参数对。
   * **解析均衡负载凭证**：提取出第一组参数，再次按照等号（`=`）进行切分，取得右侧的 **`openstack_cookie_insert`** 凭证值。
   * **解析会话会话令牌**：提取出第二组参数，同样按照等号（`=`）分割，取得右侧的 **`SESSION`** 凭证值。
4. **回写状态变量定义**：
   * 获取当前最新的系统时间，格式化输出为 `%Y-%m-%d %H:%M:%S`（更新时间）。
   * 初始化定义 **`扫码次数 = 0`**（扫码已成功，重置计数）。
   * 初始化定义 **`失效时间` 为空值**（Cookie 已更新，清空过期标记，以便客户端下一次可以直接放行静默登录）。

### 第二阶段：后台页面跳转与属性提取
5. **悬停触发下拉菜单**：模拟鼠标悬停操作，将鼠标指针悬停至页面右上角的用户名标签 `//span[@class='username']` 上，触发系统菜单下拉展现。
6. **点击进入个人账号设置**：模拟鼠标点击下拉菜单中的 `"账号设置 "` 链接。
7. **挂起等待页面渲染**：执行 2 秒延迟，随后进入条件循环，持续检测 `"用户账号"` 标签 `//label[text()='用户账号']` 是否在页面上成功渲染，直到加载完毕后才放行。
8. **提取省账号并进行防空校验**：
   * 爬取用户账号文本节点 `//label[text()='用户账号']/../div/p` 的内容，写入变量 `省账号`。
   * 使用 `while 省账号 == ''` 循环进行空值拦截，确保必须抓取到有效的非空省账号文本，防范因网页异步延迟导致的空数据写入。
9. **提取组织机构并进行防空校验**：
   * 同理，爬取组织机构文本节点 `//label[text()='组织机构']/../div/p` 的内容，写入变量 `组织机构`。
   * 使用 `while 组织机构 == ''` 循环进行空值拦截校验，确保必须拿到有效的真实组织名。
10. **正常跳出大循环并返回**：
    * 成功获取到全部凭证与实名信息后，执行 `Break` 破出外层的 For 重试循环。
    * 调用返回组件将 `SESSION`、`openstack_cookie_insert`、`扫码次数`（0）、`失效时间`（空）、`更新时间`、`省账号` 和 `组织机构` 变量完整打包，回传给服务端主流程。
