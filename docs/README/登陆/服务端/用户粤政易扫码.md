# 用户粤政易扫码子流程-核心业务理解与流程拆解

> - **本篇重点**：聚焦于服务端核心子流程 `用户粤政易扫码` 的执行细节，分析如何通过 Chrome 浏览器控制、安全二维码截图回写以及轮询判定扫码结果的完整业务闭环。

---

## 1. 为什么需要这个子流程？（设计意图与架构定位）

本流程属于服务端的关键核心子流程。当前台页面无法获取有效登录凭证时，主服务会直接调用本子流程：

* **人机交互代理**：由它充当“虚拟浏览器操作员”，在独立的服务器环境打开省系统页面，模拟人去触发登录操作，抓取二维码并传送给远端的业务人员。
* **状态侦听网关**：它不仅要发送二维码，还要“守在网页前”盯着登录页面的元素状态，直到用户通过手机扫码成功（或扫码超时），并把这个状态及时反馈给主调度流程。

---

## 2. 文字描述的扫码流程执行过程

扫码子流程的运行过程可以拆解为四个主要阶段，具体流转逻辑如下：

### 第一阶段：浏览器跳转与登录准备
1. **点击已有绑定登录**：机器人检测前台 Chrome 浏览器窗口，检查是否存在 XPath 为 `//a[text()="已绑定，点击马上登录"]` 的按钮。如果存在，则模拟鼠标执行点击（Block 1 ~ Block 2）。
2. **失败兜底刷新**：若上述点击操作返回假（失败），流程进入 Else 条件：执行 `Browser.refresh` 刷新网页，等待 2 秒后，通过 JavaScript 脚本强制执行对该登录按钮的点击，以确保能够拉起扫码界面（Block 3 ~ Block 5）。
3. **二维码渲染检测**：检测页面 iframe 中是否出现了 Class 名包含 `qrcode lightBorder` 的二维码图片元素（Block 9 ~ Block 10）。
4. **强行跳转重定向 URL**：如果上述检测判定二维码并未加载成功，系统会在 2 次循环内尝试重新点击；如果仍未加载，则直接控制浏览器跳转（navigateURL）到省系统的统一身份认证登录页（Block 11 ~ Block 18）。

### 第二阶段：二维码截图与 Socket 传输
5. **定位并截取元素图片**：精确定位到 iframe 下的 `//img[contains(@class,'qrcode lightBorder')]` 元素，执行**网页元素局部截图**，获得图片的 Base64 编码大文本。
6. **保存至本地物理路径**：将该二维码图片同步输出并保存到服务端机器的本地物理路径下，以当前时间命名。
7. **Socket 字节流回传**：执行 Python 代码块，将 Base64 二维码字符串进行 UTF-8 编码，通过已建立的 Socket TCP 物理链路（`conn_socket.sendall`），将其直接灌回给正在等待的客户端，触发前端的分发与展示。

### 第三阶段：多终态扫码结果轮询
8. **哨兵变量初始化**：设置控制轮询周期的四个基础变量：`首页 = False`、`等待次数 = 0`、`红叉 = False`、`多账号选择 = False`。
9. **开启轮询检测循环**：开启常驻的 `while` 循环，每次循环体执行前通过 `sleep` 延迟 2 秒。最大允许轮询 8 次（即共 16 秒的手机扫码等待周期）。
10. **分支走向 A（扫码失败/失效检测）**：
    * 检测页面上是否存在 XPath 为 `//*[@id='loginFailed']` 的红叉失败元素。
    * 如果显示该元素，说明用户取消了扫码或二维码已失效。流程更新变量 `红叉 = True`，并执行 `Control.break` 直接破出循环（进入异常收尾）。
11. **分支走向 B（多组织机构账号选择）**：
    * 检测页面是否出现了 `//div[text()='选择要登录的省事项管理系统账号']` 容器元素。
    * 如果出现，判定该用户账号绑定了多个组织。机器人自动定位到参数指定的 `组织机构` 对应的输入多选框 `//span[text()='%s']/parent::td/preceding-sibling::td[3]/input` 并执行点击。
    * 接着点击下方的 `"马上进入省事项管理系统"` 按钮提交登录。
    * 随后检测网页是否显示 `//span[text()='首页']`。若显示，更新 `首页 = True` 并破出循环。
12. **分支走向 C（直接成功检测）**：
    * 如果页面没有显示多选账号，直接检测是否存在 `//span[text()='首页']` 元素。如果检测成功，说明用户在手机端授权后直接登入，更新变量 `首页 = True`。
13. **等待计数累加**：如果以上分支均未命中，系统执行 `等待次数 = 等待次数 + 1`，继续下一轮 2 秒挂起等待，直到 16 秒超时。

### 第四阶段：状态返回
14. **流程返回与主控衔接**：调用 `Control.return` 将布尔值 `首页` 结果回传给主流程。如果登录成功，主流程将启动后续的 `获取对应省信息` 流程来提取 Cookie。
