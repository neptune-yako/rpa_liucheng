# 省系统扫码登录服务端-核心业务理解与流程拆解

> - **本篇重点**：聚焦于服务端主控制流程 `省系统扫码登录服务端` 的执行细节，分析 Socket 常驻服务生命周期、多分支逻辑分发、MySQL 数据回写以及第三方回调 API 通信的业务流转机制。

---

## 1. 为什么需要服务端主流程？（设计意图与架构定位）

本流程是整个扫码登录体系的“**后端中枢大脑**”。它与前台的《中间件》和远端的《客户端》配合工作，扮演着控制与协作的核心：

* **套接字守护进程**：在服务器后台常驻，并独占绑定一个分配的服务端端口（如 `13002`）。它保持挂起状态，专门用于接收客户端发起的登录申请。
* **业务流程协调者（Orchestrator）**：接收到客户端的任务账密包后，它连接数据库获取用户的状态，并根据状态决定何时调用《用户粤政易扫码》拉取二维码、何时调用《获取对应省信息》收割凭证、何时对数据库执行写入。
* **第三方接口桥梁**：无论扫码最终是成功还是失败（用户未扫、无绑定账号等），它都负责向外部的统一身份认证管理平台发起 HTTP POST 登录回调，将结果实时通知外部业务门户。

---

## 2. 文字描述的流程执行过程

主服务在启动后会经历“前置初始化”和“常驻业务循环”两个阶段，具体的执行逻辑如下：

### 第一阶段：前置初始化与 Socket 绑定
1. **套接字绑定与侦听**：主流程启动时，首先在后台创建 TCP Socket 对象，将其绑定至本地回路 IP `127.0.0.1` 和输入的参数 `端口号`（如 `13002`）上。调用侦听方法，设置挂起排队连接队列的最大长度为 5。
2. **开启无限大循环**：开启常驻的条件循环判断，整个业务周期被完全包含在此循环中。外层通过 `Try-Catch-Finally` 结构进行全面保护，确保在任何时刻发生流程报错时，都能进入 Finally 块释放系统资源。

### 第二阶段：常驻业务循环与客户端交互
3. **打开浏览器容器**：在每次循环开始时，自动调起 Chrome 浏览器，跳转至省事项管理后台的主入口，并自动加载扩展插件。
4. **挂起等待客户端连接**：执行内置 Python 脚本，调用阻塞式的套接字接受方法。此时流程挂起，直至收到客户端的 TCP 连接。
5. **读取并解析任务字典**：连接建立后，从连接中读取最大 4096 字节的数据包，解析出客户端投递的 JSON 任务字典，并将其解包为 `用户账号`（useraccount）和 `组织机构`（department）变量。
6. **建立数据库连接**：机器人连接至 MySQL 数据库 `亿讯-登录系统`。
7. **检索用户历史记录**：执行 SQL 语句，查询该用户在 `用户账号对应表` 中的历史 Cookie 时效及扫码状态。
8. **逻辑分发判定（三大处理分支）**：
   * **分支 A：新用户处理分支（在库里查无记录）**：
     * 初始化该用户的本地扫码次数为 0，随后调用《用户粤政易扫码》子流程。该子流程会自动截图二维码并通过 Socket 发送给客户端，返回扫码判定结果。
     * 服务端随后将本地 `扫码次数` 累加 1。
     * **若判定为红叉（无省账号）**：向上游回调接口发送 POST 登录失败请求（code: "0"，描述为“无绑定省账号”）。
     * **若判定为成功（扫码进入首页）**：调用《获取对应省信息》抓取最新 Cookie 并清空扫码次数，向上游发送 POST 登录成功请求（code: "1"），最后通过 SQL 执行对该账号的 `INSERT` 插入更新。
     * **若判定为扫码超时/未扫码**：清空临时凭证，发送登录失败请求（code: "0"，描述为“用户未扫码”），并将初始失败信息以 SQL 执行入库。
   * **分支 B：已存老用户且当前扫码次数为 0 分支**：
     * 校验失效时间。若失效时间不为空（过期），则与“新用户”类似，调用《用户粤政易扫码》子流程下发二维码，并在用户扫码成功后，调用《获取对应省信息》收割凭证，通过 `UPDATE` SQL 更新失效时间为 null，同时向外部平台发起 POST 成功回调（code: "1"）。若用户未扫码，则向外部平台发起 POST 失败回调（code: "0"）。
   * **分支 C：登录中的老用户分支（扫码次数在 1 ~ 1000 次之间）**：
     * 将当前在库中的扫码次数读取出来并**累加 1**。
     * **先写库再扫码**：立即执行 `UPDATE` SQL 语句，将累加后的扫码次数更新至数据库。这一步非常关键，它确保了每一次发起扫码未成功的状态都能被数据库实时记录。
     * 接着调用《用户粤政易扫码》子流程，下发二维码并等待扫码结果。
     * 若扫码成功，再次调用《获取对应省信息》子流程，重新将数据库中的 `扫码次数` 重置归零（code: "1" 的成功回调）。若扫码失败或超时，则发送 code: "0" 的失败回调，不重置次数。

### 第三阶段：收尾与环境清理 (Finally)
9. **全局异常捕获处理**：如果在上述运行过程中（如数据库连接断开、网络闪断）抛出任何异常，流程将捕获错误日志，并向连接的客户端 Socket 强制回传错误报文 `"流程运行异常"`，防止客户端挂起死等。
10. **Finally 块强制资源释放**：无论本轮任务是执行成功还是异常报错，流程最后必将进入 Finally 块，依次执行以下物理清理：
    * 关闭 MySQL 数据库连接，释放连接池。
    * 关闭前台 Chrome 浏览器窗口，释放内存。
    * 删除浏览器留存的全部 Cookie。
    * 将浏览器重定向回 cas 认证根路径，为下一轮循环做好净空准备。
