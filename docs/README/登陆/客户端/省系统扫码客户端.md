# 省系统扫码客户端-核心业务理解与流程拆解

> - **本篇重点**：聚焦于以文字描述的方式，理解该 RPA 机器人在整体架构中的定位、设计意图以及完整的业务逻辑流转过程。

---

## 1. 为什么需要这个机器人？（设计意图与“包装”概念）

在政务事项的自动化录入与推送中，最核心的障碍是**“省级系统要求必须使用手机进行粤政易扫码认证才能登录”**。如果直接运行后台数据推送脚本，由于安全限制，同步会彻底中断。

为了在“保障安全认证”与“实现自动化批量处理”之间取得平衡，系统在真正的网页登录前**包装（Wrap）了一层拦截与缓冲网关**，即本项目：

* **解耦（业务异步化）**：客户端通过任务队列，把实时的、阻塞式的“前台网页扫码需求”转变为可以异步回填、包含 Base64 截图数据的“后台任务报文”。
* **静默化（免除重复扫码）**：每次推送前，客户端先查询数据库，只要库里存有的 `SESSION` cookie 仍有效，就完全跳过扫码过程直接放行，直接拿 Cookie 静默同步，保障无人值守自动运行。
* **按需自动分发**：只有当 Cookie 确实缺失或失效时，才会向服务端申请并抓取二维码，自动分发给网页端的具体经办人扫码，最大程度减少人工干预。

## 2. 文字描述的 RPA 流程执行过程

整个【省系统扫码客户端】在实际的扫码登录外层起到“网关拦截与分发”的作用。具体运行过程如下：

1. **实时监听与监控**：流程后台运行并开始监控控制台的任务队列（类似于 MQ 消息中间件队列，对应任务队列名称为“市政数登录系统”）。
2. **实时消费队列任务**：一旦发现队列中有待同步的任务，机器人立即实时消费并抓取该任务的数据报文。
3. **获取数据并定义变量**：提取任务包中的信息，将其中的 `useraccount` 赋值给本地变量 `用户账号`，`department` 赋值给本地变量 `组织机构`，为后续的流程决策和数据填写做准备。
4. **建立数据库连接**：机器人使用硬编码凭据，连接到远程的 MySQL 数据库（`10.196.193.23` 端口 `3306` 上的 `亿讯-登录系统`）。
5. **查询数据库中的用户信息**：执行 SQL 语句，查询该“登录用户”账号在库里的历史记录：
   ```sql
   select SESSION, openstack_cookie_insert, 省账号, 扫码次数, 失效时间 
   from 用户账号对应表 
   where 用户账号 = '{用户账号}'
   ```
6. **新老用户状态初判**：判断查询结果列表 `result` 是否为空（即 `result == []`）。
7. **分支走向 A：判定为新用户**：
   * 如果 `result` 为空，说明数据库无记录，直接进入新用户扫码路线。
   * 机器人通过 Socket 向本地中间件 `127.0.0.1:13001` 发送 `"请分配一个端口"` 并获取分配的临时服务端端口（如 `13002`）。
   * 接着建立与该服务端端口的 TCP 连接，将完整的任务字典投递出去。
   * 阻塞等待服务端回传二维码的 Base64 截图，收到截图后，向控制台回填任务状态为 `3`（标记 `Scan: "True"`, 并附带 `QRcode` 截图数据），提示用户在前端扫码，流程在此结束。
8. **分支走向 B/C/D/E：判定为已存老用户**：
   * 如果 `result` 不为空，则说明数据库中已有该用户的历史状态，开始进行后续的策略判定。
9. **校验省账号绑定状态（路线 B）**：
   * 检查拉取到的字段，判断该账号的 `省账号` 是否等于 `'无对应省账号'`。
   * 如果是，说明该账号由于未绑定或被系统锁定，直接停止后续流程，向控制台回填状态为 `3`（标记 `Scan: "False"`, 返回“用户没有对应省账号，不需要扫码”原因），备注为 `"不需要扫码噢"`。
10. **进入扫码次数与 Cookie 校验期**：
    * 如果 `省账号` 正常，则将读取到的 `result[0]['扫码次数']` 提取出来，准备做熔断判定和时效判定。
11. **取出失效时间字段**：
    * 提取 `result[0]['失效时间']` 字段的值，作为 Cookie 时效性的直接判定依据。
12. **执行 Cookie 时效判定（路线 C 与路线 D）**：
    * **如果扫码次数为 0 且失效时间为 `None` 或 `''`（空值）**：代表用户的 Cookie 仍在有效期内（路线 C1），直接停止流程并回填状态码 `3`（标记 `Scan: "False"`, 提示“用户cookie有效，不需要扫码”）。
    * **如果失效时间不为空（过期）或扫码次数在 1~1000 次之间**（路线 C2、路线 D）：说明 Cookie 已过期，必须重新扫码。此时流程与“新用户”类似，连接中间件获取服务端端口，建立 Socket 连接向服务端发送任务字典并阻塞获取最新的二维码截图，随后回填状态码 `3`（标记 `Scan: "True"`）要求前台用户扫码。
13. **流程的正常与异常终止出口**：
    * **【出口 1：成功扫码】**：用户在前台看到二维码并使用手机扫码成功后，服务端自动抓取新 Cookie 写入数据库，更新 `SESSION` 与 `openstack_cookie_insert`，并将 `失效时间` 抹空（抹空后，下次检测就会命中第 12 步的有效分支直接跳过），同时**重置 `扫码次数` 为 `0`**。
    * **【出口 2：锁定熔断（失败停下）】**：若用户一直不扫码或扫码失败，服务端会通过 SQL 将数据库中的 `扫码次数` 累加 1。当扫码失败次数不断累加达到 **`1001` 次**时（即失败 1000 次），客户端在此判定并直接触发路线 E。路线 E 执行 SQL 将 `省账号` 修改为 `'无对应省账号'` 并抹除 Session。锁号后，该账号后续的任务进来时会在第 9 步（路线 B）被直接拦截退回，再也不会弹出二维码，强行停止循环空转。

## 3. 关键运营 Q&A

### Q1: 一共有哪些控制台状态码？为什么异常和无需扫码都回填“3”？
任务状态数据回填节点（`TaskDataItem.setDataStatus`）中定义了 7 个标准状态值：
* **`1`** (等待执行) / **`2`** (正在执行)
* **`3` (执行成功)** / **`4`** (执行失败)
* **`5`** (等待超时) / **`6`** (已删除) / **`7`** (重试执行)

**核心机制**：本流程中，只要客户端完成了它的逻辑流转（成功生成并分发了二维码，或者判断出无需扫码，甚至在 `try-catch` 中捕获到异常），客户端都会将该队列任务回填为 **`3`（执行成功）**。
* **原因**：如果回填失败（4）或超时（5），控制台的重试报警会打断后台批量任务的正常推进。将二维码发送至外部展示或异常信息回传给业务层后，本步骤调度已安全收尾，应标识为“3”放行，由后面的具体业务节点处理状态。

---

### Q2: 熔断锁定的账号如何解除？
当同一个账号连续失败/未扫码达到 1001 次后，其 `省账号` 字段会在数据库中被强行覆盖为 `'无对应省账号'`。
* **解锁方案**：必须由运维或技术开发人员直接在 MySQL 数据库的 `用户账号对应表` 中，手动将该账号的 `扫码次数` 重置为 `0`，并将 `省账号` 修改回正常绑定的账号值。
