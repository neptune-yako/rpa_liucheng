# 根据名称分组新增任务数据-核心业务理解与流程拆解

> - **本篇重点**：聚焦于主控数据分发流程 `根据名称分组新增任务数据` 的执行细节，阐明其控制台标识监听机制、10小时超时熔断重置、双数据源读取、Pandas 事项分组排序以及任务队列打包推送的完整闭环。

---

## 1. 为什么需要这个流程？（设计意图与架构定位）

本流程是数据推送体系中的“**任务打包与分流总控（Task Packager）**”。它直接决定了什么时候从事项数据源中抓取数据，以及怎么把海量无序的数据整合成时序正确的任务：

* **控制台触发开关**：采用“变量触发”方式，只有当控制台中的标识变量被开启时才执行新增。
* **时序依赖保护（Pandas 分组排序）**：在政务录入中，有些事项数据存在强时序依赖（例如同一个事项必须“先新增后修改”）。如果数据杂乱无序地投递到队列中，多机器人并发执行会导致业务逻辑错乱。因此，本流程利用 Pandas 将数据按“事项名称”和“组织机构”进行组内升序排序，保证队列中同一事项任务的执行顺序绝对符合时间轴。
* **时序异常检测与过滤**：在打包过程中，若发现同一组数据中包含了以前运行异常的记录（转录状态为 `'9'`），为了防止队列被污染或反复报错卡死，会智能跳过这组事项的同步。

## 2. 文字描述的流程执行过程

整个主控流程在后台常驻运行，具体步骤拆解如下：

### 第一阶段：控制台标识轮询与 10 小时超时熔断
1. **启动测试日志**：流程启动时首先打印日志，记录当前运行的版本（测试版本为 `20250510`）。
2. **进入常驻监听死循环**：开启 `while True` 的无限大循环。
3. **轮询频率控制**：在循环体的最开头执行 30 秒的静默延迟，即系统每隔 **30 秒** 对控制台状态执行一次轮询。
4. **获取标识变量**：调用 `VariableMgt.selectVariableMgt` 获取控制台配置的全局变量 **`"是否新增任务数据"`**，并使用 JSON 解析组件将其转化为 Python 字典。
5. **分支走向一（空闲挂起）**：
   * 检查字典中的 `'新增标识'` 键值是否等于 `0`。
   * 如果等于 `0`，说明当前没有数据同步需求，流程在打印“不需要新增任务数据”后，执行 `Control.continue` 跳过本轮循环，继续处于监听挂起状态。
6. **分支走向二（超时熔断防卡死）**：
   * 如果 `'新增标识'` 等于 `1`，系统会提取字典中的 `'设置时间'` 字段，并获取当前系统时间。
   * 使用时间计算组件以 **分钟（m）** 为单位计算两者的时差。
   * **若时间差大于 600 分钟（即 10 小时）**，判定为上一次开启后发生了卡死或控制台未能及时复位。流程会自动生成待传字典 `{'新增标识': 0, '设置时间': 当前时间}` 并转化为 JSON 字符串，调用变量更新接口强行将 `"是否新增任务数据"` 复位设为 `0`，并执行 `Control.continue` 结束本轮循环，防止系统无限空转。

### 第二阶段：多数据源读取与清洗子流程调用
7. **建立数据库连接**：如果 `新增标识` 为 1 且未超时，流程开始连接事项推送数据源数据库。代码中配置了两个连续的 MySQL 连接节点（Block 28 & Block 29），最终连接生效的目标为：主机 `10.194.252.211` 端口 `8066`，用户名 `rpaQuery`，密码 `rpa123!`，数据库名 `rpa_query_db`。
8. **查询未同步记录**：执行 SQL 语句查询待录入的数据记录：
   ```sql
   select * from rpa_item_province_push 
   where status not in ('N','1') 
   order by create_time, obj_type, oper_type
   ```
   并将查询出来的记录命名为 `数据库全量数据`。
9. **调用清洗子流程**：调用子流程 **`3jhlazJo`（生成数组.json）**，传入 `数据库全量数据` 和当前的数据库连接对象，获取转换清洗后的 `待写入数组`。

### 第三阶段：Pandas 分组与时序排序
10. **构建 Pandas 数据帧**：在自定义 Python 代码块中，将 `待写入数组` 排除首行标题后，加载至 Pandas 的 `DataFrame` 中。
11. **按“事项与组织”联合分组**：执行 `df.groupby(['事项名称','组织机构'], sort=False)` 对数据帧按事项和部门进行物理分组，并使用 For 循环遍历这些分组。
12. **组内强时序排序**：对每一个分组内的记录集，按照 `['创建时间','操作对象','操作类型']` 进行**全升序（Ascending=[True, True, True]）排序**。这确保了该事项的流水日志完美符合创建时间先后。
13. **异常与未就绪过滤（双重拦截）**：
    * **拦截 1（包含异常 9）**：提取该分组的 `转录状态` 列表。如果列表中包含 `'9'`（历史同步异常状态），直接打印“有异常，该事项不同步”并跳过整个分组。
    * **拦截 2（无待录入记录 0）**：如果列表中不包含 `'0'`（代表这一组中没有任何待同步记录），打印“没有需要同步的记录”并跳过该分组。
14. **强转字符串类型**：对于通过拦截的安全数据分组，将创建时间、更新时间、转录状态、错误原因、事项名称、组织机构字段统一强制转换为 `string` 类型，避免格式引发报错。

### 第四阶段：任务投递与状态复位
15. **投递至任务队列**：将排序转换后的分组数据调用 `.to_numpy().tolist()` 转换为 Python 的二维列表。调用任务数据添加接口（`TaskDataItem.addData`）投递至控制台队列 **`“亿讯二次录入推送数据”`**，报文负载为 `{'推送数据': 列表}`。
16. **Excel 导出子流程（已禁用）**：调用子流程 `xYMUXGeM`（写入excel.json），但在 main 流程中，该 Block 的启用标识 `is_enable` 和 `self_enable` 被明确配置为 **`False`**（在生产运行中处于**禁用静默状态**，不会执行 Excel 本地导出）。
17. **释放数据库连接**：调用 MySQL 关闭连接接口关闭 `conn` 连接。
18. **重置控制台变量（归零）**：成功投递所有数据后，流程获取当前系统时间，构建字典 `{'新增标识': 0, '设置时间': 当前时间}` 转化为 JSON，调用变量写入组件**将控制台变量 `"是否新增任务数据"` 重置为 0**，使整个主控重新挂起，结束本轮数据处理生命周期。
19. **全局异常捕获**：在 Catch 块中捕获可能产生的异常，并将其转换为字符串打印输出。
