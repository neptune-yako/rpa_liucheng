# 市政数事项主调度与子流程路由-核心业务理解

> - **本篇重点**：聚焦于主调度分发中枢 `市政数事项` 的核心流转逻辑，阐明其前后台自我续航机制、Cookie 免密登录与政数局特例路由、52个子流程的决策路由矩阵以及异常截图熔断的闭环设计。

---

## 1. 架构定位与业务意图

本流程是整个事项同步体系中的“**总调度分发中枢（Dispatcher）**”。它常驻后台运行，专门负责消费新增队列中的待同步记录，并作为控制大脑，根据流水的业务类型，挑选对应的功能积木（子流程）在 Chrome 浏览器中模拟前台填报。

---

## 2. 主调度流程核心流转逻辑

主调度流程在运行期间，核心逻辑步骤可以归纳为以下五个主要阶段：

### 第一阶段：常驻监听与前后台自我续航机制
1. **获取工作路径**：流程启动时自动检测 RPA 部署所在的盘符，在全局变量中定义 `glv['文件保存路径'] = rf'{disk}\文件保存'` 用于后续保存异常截图。
2. **队列消费监控**：常驻死循环运行，消费队列 **`“亿讯二次录入推送数据”`**。
3. **空闲队列激活（自我续航）**：
   * **若消费到的任务数据为空（队列中无任务）**：主调度会查询控制台的全局变量 `"是否新增任务数据"`。
   * 如果该变量的 `'新增标识'` 为 `0`（代表前台新增程序处于闲置状态），主调度会**主动向控制台写入新值将其改为 `1`**。
   * 这将直接唤醒处于挂起状态的“前台数据打包流程”去拉取下一批流水并塞入队列。主调度随后静默 60 秒后重试，实现流程的全自动流转与闭环。

### 第二阶段：账号 Cookie 注入与“政数局”绿灯特例
4. **定位部门 Cookie**：当成功消费到任务后，主流程反查操作人账号的归属组织，并在本地库中检索该组织在有效期内的 Session 会话。
5. **市政数局免切换通道**：
   * **特例判断**：若流水的目录层级为 `'4'`（区级主项子项政策兑现或公共服务目录等类目）或主系统同步标识 `sync_province` 为 `'0'`。
   * **账号重定向**：主控会绕过原部门，强制调用 **`'szsj_admin'`（市政数局账号）** 的 Session，使用该全局特权账号完成统一同步，免去频繁切换部门账号带来的交互消耗。
6. **免密登录系统**：调用子流程 `机器人账号登录.json` 将 Cookie 会话（SESSION 和 openstack_cookie_insert）直接注入当前浏览器，实现免密进入系统首页。

### 第三阶段：分类决策与子流程路由分发
7. **数据前置拦截**：
   * 检查该记录的转录状态。若在数据库中查询出已处于成功 `'1'` 或废弃 `'N'`，直接 `continue` 跳过。
   * 检查该记录是否已被其他机器人锁定执行（`status == '2'`）或异常（`status == '9'`），执行对应过滤与任务删除。
8. **核心路由路由**：读取记录中的 `obj_type`（操作对象）和 `oper_type`（操作类型），将其分流给对应的子流程（详见第 3 节的决策矩阵表）。

### 第四阶段：异常处理与现场截图 (Catch)
9. **故障截图留痕**：如果在子流程填报过程中出现任何网络假死、元素未找到或系统闪退等异常，主控会捕获错误，并立刻对浏览器当前屏幕进行截图，以时间戳命名保存至本地：`{文件保存路径}\{时间戳}.png`。
10. **错误状态上报**：在 Catch 中将具体报错信息整理后追加至异常变量。

### 第五阶段：数据库状态回填与任务大失败熔断 (Finally)
11. **单条结果回写**：单条数据录入完成后，无论成功还是失败，均会进入 Finally 块，首先调用子流程 **`ohsC0p3a`（更新数据库.json）**，将该流水的处理状态（如 3成功、4失败）和错误原因更新到 MySQL 的原始流水表中。
12. **任务整体熔断拦截**：
    * **失败断流**：若单条数据同步判定为 False（失败），主控会调用接口**把整个大任务的状态标记为 4 (失败)** 并执行 Break 破出循环，停止执行后续事项，防止脏数据雪崩或重复卡死。
    * **全部成功**：若列表中所有事项均顺利同步完毕，循环正常退出，则在循环外**将整个任务状态标记为 3 (成功)**。
13. **释放与防高频卡死**：关闭 Chrome 浏览器、释放数据库连接。若单轮运行时间不足 60 秒，则延迟补足至 60 秒，防止频繁发送请求压垮控制台。
