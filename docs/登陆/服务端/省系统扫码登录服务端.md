# 省系统扫码登录服务端-交付与运营全景文档

> - **文档最后更新时间：** 2026-06-30

## 1. 系统概述与运行环境
- **1.1 项目背景**：本流程为常驻后台的服务端机器人，主要负责开启指定的 Socket 监听端口，接收客户端分配来的登录请求任务。它通过谷歌浏览器自动化打开省系统页面，配合扫码子流程及信息抓取子流程完成用户登录状态的验证，将提取到的 Cookie (SESSION) 写入数据库并回调第三方系统通知接口。
- **1.2 人机协同与触发机制**：本流程为后台服务，一旦启动则在本地持续循环侦听。当有客户端发起连接时被动触发。登录过程中通过调用子流程进行网页操作及二维码捕获，后续抓取信息全自动运行。
- **1.3 技术选型与依赖组件**：
  - **工程版本**：V8.7.7
  - **主要逻辑模块**：
    * 主控流程：省系统扫码登录服务端.json
    * 子流程（抓取信息）：获取对应省信息.json
    * 子流程（扫码）：用户粤政易扫码.json
  - **核心组件**：RPA 浏览器组件 `Browser`（加载谷歌浏览器插件 `/BrowserPlug/SD-RPA.crx`）、数据库连接模块 `mysql`、内置 Python 代码块。
  - **网络通信**：本地回路 TCP Socket 服务侦听。

## 3. 正常作业流程 (SOP)
- **3.1 步骤一：任务投递/触发阶段**：
  - 启动服务端流程后，它将绑定本地 `127.0.0.1` 指定端口（默认 `13002`），调用 `sock.accept()` 形成阻塞等待。
  - 客户端连接后，接收 JSON 格式的 `任务数据字典`，解析出 `用户账号` 和 `组织机构`。
- **3.2 步骤二：核心处理/交互阶段**：
  - 流程连接数据库并查询 `用户账号对应表`：
    - **新用户（无记录）或扫码次数为 0 的用户**：
      1. 调用子流程 `用户粤政易扫码` (`OgYcI2gg`) 以截取二维码发送给客户端并等待扫码。
      2. **若扫码成功**（`首页 == True`）：主流程继续调用子流程 `获取对应省信息` (`Er2ayzBa`) 以抓取最新 Cookie 并获取用户在省系统的真实信息。
    - **扫码次数在 1 ~ 1000 次之间的用户**：
      1. 流程首先执行 SQL 更新语句，将 `用户账号对应表` 中对应用户的 `扫码次数` 累加 `1`，同时更新 `更新时间`。
      2. 调用子流程 `用户粤政易扫码` 并根据返回的 `首页` 结果决定是否进行 Cookie 抓取。
- **3.3 步骤三：结束与回填阶段**：
  - **登录成功**：
    - 调用子流程 `获取对应省信息` 抓取并分割 Cookie 获得 **`SESSION`** 和 **`openstack_cookie_insert`**，重置 `失效时间 = ''`，`扫码次数 = 0`。
    - 向回调接口发送 POST 请求（统一回调接口：`http://10.194.252.94:8090/gsimGateway/gsim-web/scanCode/scanCallBack`），回传 JSON 报文：`{"userName":"{用户账号}", "code":"1", "desc":"登录成功"}`。
    - 执行 SQL 写入/更新表结构：
      ```sql
      insert into 用户账号对应表(用户账号,省账号,SESSION,openstack_cookie_insert,更新时间,扫码次数,组织机构) values('%s','%s','%s','%s','%s','%d','%s') ON DUPLICATE KEY UPDATE SESSION = VALUES(SESSION),openstack_cookie_insert = VALUES(openstack_cookie_insert),更新时间= VALUES(更新时间)
      ```
  - **扫码超时/用户未扫码**：
    - 向回调接口回传登录失败报文：`{"userName":"{用户账号}", "code":"0", "desc":"登录失败，用户未扫码"}`。
    - 执行相同的 SQL 插入/更新表结构，以保存当前累加的扫码次数及失效状态。
- **3.4 重入与并发规则说明**：
  - **重试性**：支持重试，每次重试均会产生新的 Socket 交互和扫码次数计算。
  - **并发性**：通过端口映射实现并发控制。每个服务端实例独占一个端口侦听连接。需要根据客户端传入的配置，独立开启多个服务端实例并分别绑定不同的端口号。

## 4. 业务异常提示与人工干预指南
- **4.1 业务拦截异常处理**：
  - 若子流程判定用户扫码无效（`红叉 == True`），代表该用户账号在省系统中未绑定，主流程会向回调接口回传失败报文：`{"userName":"{用户账号}", "code":"0", "desc":"登录失败，该用户没有绑定省账号"}`，不更新 Cookie 数据。
- **4.2 熔断/超时异常处理**：
  - 熔断和黑名单保护逻辑主要在客户端完成。当服务端收到连接时，若数据库中用户的扫码次数已经超限，流程不予以登录处理。

## 5. 数据库设计与状态机
- **5.1 核心表结构说明**：
  - 连接数据库 `亿讯-登录系统` 中的 `用户账号对应表`。
- **5.2 状态字段映射**：
  - 主流程依赖子流程 `获取对应省信息` (`Er2ayzBa`) 在浏览器内提取的变量：
    - `SESSION`：获取到的省系统会话 Cookie。
    - `openstack_cookie_insert`：获取到的负载均衡 Cookie。
    - `省账号`：获取的真实省系统用户账号（通过点击网页 `//a[text()='账号 ']` 链接后，在元素 `//label[text()='用户账号']/../div/p` 提取）。
    - `组织机构`：获取的省系统组织机构名称（在元素 `//label[text()='组织机构']/../div/p` 提取）。

## 6. 常见技术故障排查（FAQ）
- **6.1 网络与通信故障**：
  - **现象**：绑定侦听端口失败（端口被占用）。
  - **排查命令**：
    1. 在本机命令行使用 `netstat -ano | findstr {端口号}` 查看占用进程的 PID。
    2. 使用进程管理工具或控制台结束占用端口的残留进程。
- **6.2 组件与数据异常**：
  - **现象**：网页加载极慢或提示无法访问。
  - **方案**：确认本地连接的远程页面 `https://19.15.0.100/...` 网络未断开。如因页面假死，流程捕获异常后将通过 `finally` 自动执行 `Browser.closeBrowser` 强行释放浏览器句柄。

- **7.3 磁盘与缓存清理**：
  - 流程在处理扫码无效和未扫码时，仅发送回调报文与执行少量 SQL，对磁盘空间的压力较小。仅需对引擎本地日志文件进行定期归档与清理。
