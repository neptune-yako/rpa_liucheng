# 省系统扫码登客户端-交付与运营全景文档

> - **文档最后更新时间：** 2026-06-30

## 1. 系统概述与运行环境
- **1.1 项目背景**：本流程为后台服务型机器人，用于自动消费指定队列的任务数据，检索本地数据库中对应用户账号的 Cookie 状态与扫码记录。当用户需要扫码时，通过 TCP Socket 建立通信以拉取最新的登录二维码大文本（Base64 截图），最后执行任务数据的状态回填，以此来辅助管理省系统的登录状态。
- **1.2 人机协同与触发机制**：本流程由控制台任务队列自动触发。在用户 Cookie 记录有效、或被标记为无省账号时，流程全自动运行；若需要扫码，流程会通过接口回填二维码的 Base64 数据并等待人工处理。
- **1.3 技术选型与依赖组件**：
  - **流程版本**：V1.7.4.5
  - **核心依赖**：RPA 内部模块（内置数据库连接模块 `mysql`、通信库、流程控制 `Control`）
  - **数据库**：MySQL 数据库（连接变量为 `conn`）
  - **网络通信**：TCP Socket 连接（本地回路 `127.0.0.1` 端口 `13001`）

## 2. 运行前准备
- **2.1 系统与环境检查**：
  - 确认运行机器至本地 TCP 回路 IP `127.0.0.1` 端口 `13001` 的网络连接可用（用于连接中间件进行端口分配）。
  - 确认运行机器可直接连接 MySQL 数据库服务：`10.196.193.23:3306`，数据库名：`亿讯-登录系统`。
- **2.2 软硬件就绪确认**：
  - 必须确保后台分配中间件及相关服务端端口正常侦听且没有冲突。
- **2.3 安全与凭据合规说明**：
  - 流程直接通过内置组件连接 MySQL，数据库连接参数硬编码在流程块中（用户名为 `root`，密码为 `Sd_12345`，数据库名为 `亿讯-登录系统`）。应确保连接文件与代码的访问控制权限。

## 3. 正常作业流程 (SOP)
- **3.1 步骤一：任务投递/触发阶段**：
  - 任务消费节点会自动在控制台消费名称为 **“市政数登录系统”** 的任务数据，输出变量存入 `任务数据字典`，从中提取 `useraccount`（用户账号）和 `department`（组织机构）。
- **3.2 步骤二：核心处理/交互阶段**：
  - 流程连接数据库并执行 SQL 查询：
    ```sql
    select SESSION,openstack_cookie_insert,省账号,扫码次数,失效时间 from 用户账号对应表 where 用户账号='{用户账号}'
    ```
  - **流程分支处理逻辑**：
    - **分支 A（新用户）**：当数据库查询结果为空时，向 `127.0.0.1:13001` 发送 `"请分配一个端口"`，获取服务端端口号，随后向该端口发送 `任务数据字典` 并接收 Base64 截图信息 `screenshot`（缓冲区最大接收大小设为 `262144` 字节）。
    - **分支 B（无省账号用户）**：当 `省账号 == '无对应省账号'` 时，直接回填无需扫码信息。
    - **分支 C（老用户，扫码次数为 0）**：若 `失效时间` 为 `None` 或 `''`（空字符串），判定为 Cookie 有效无需扫码；否则通过 Socket 通信向服务端请求并拉取新二维码。
    - **分支 D（老用户，扫码次数在 1~1000 次之间）**：通过 Socket 通信向服务端拉取新二维码。
    - **分支 E（老用户，扫码次数 >= 1001）**：判定为多次登录失败，重置内部状态，将 `省账号` 赋值为 `'无对应省账号'`，清空 `SESSION` 和 `openstack_cookie_insert`，并通过 SQL 写入更新数据库。
- **3.3 步骤三：结束与回填阶段**：
  - **需要扫码**：调用 `setDataStatus` 回填状态码为 `3`（执行成功），回填结果为：`str({"Useraccount":"{用户账号}","Scan":"True","QRcode":"{screenshot_base64}"})`，备注为 `"请扫码"`。
  - **无需扫码**：调用 `setDataStatus` 回填状态码为 `3`（执行成功），回填结果根据情况为：`str({"Useraccount":"{用户账号}","Scan":"False","Reason":"..."})`，备注为 `"不需要扫码噢"`。
- **3.4 重入与并发规则说明**：
  - **重试性**：本流程为状态消费型，支持重入。当扫码失败或被人工清理后，再次发起该用户任务时，会重走上述查询和通信判定分支。
  - **并发性**：流程会连接本地回路固定端口 `13001` 进行端口分配，单机运行时需注意同一时间只允许一个客户端实例占用此连接，无法多实例并发抢占端口。

## 4. 业务异常提示与人工干预指南
- **4.1 业务拦截异常处理**：
  - 提示 **“用户账号没有对应省账号”**：表示用户未开通或未绑定省系统账号。处理人需到业务系统验证该用户账号的绑定状态。
- **4.2 熔断/超时异常处理**：
  - 提示 **“用户5次以上没有扫码或者没有登录成功...”**：表示同一账号的 `扫码次数` 累积达 `1001` 次及以上，流程将自动将其 `省账号` 修改为 `'无对应省账号'` 并回填 `Scan: "False"`。
  - **人工重置流程**：若需解除熔断并恢复该账号的扫码判定，需要由技术开发或数据库管理员在 MySQL 中执行更新操作，修改 `省账号` 并重置 `扫码次数`。

## 5. 数据库设计与状态机
- **5.1 核心表结构说明**：
  - 本流程使用 MySQL 远程表，涉及的表为 `用户账号对应表`（隶属于数据库 `亿讯-登录系统`）。
- **5.2 状态字段映射**：
  - `SESSION`：保存会话令牌。
  - `openstack_cookie_insert`：保存网络层 Cookie 负载。
  - `省账号`：判定该用户在省系统是否有绑定记录（拦截值为 `'无对应省账号'`）。
  - `扫码次数`：用户扫码次数，当该值达 `1001` 及以上时被判定为多次未扫码锁定。
  - `失效时间`：通过此字段的空值/None 状态判定 Cookie 是否依然有效。
- **5.3 数据交互规约**：
  * 任务回填数据格式：
    - `Scan: "True"`（代表需要扫码，携带 `QRcode` 字段）。
    - `Scan: "False"`（代表无需扫码，携带 `Reason` 字段）。
  * 流程中如果运行异常，也会强制捕获并将任务状态设置为 `3`（回填为 `Scan: "False", Reason: "异常"`），防止外部系统解析失败。

## 6. 常见技术故障排查（FAQ）
- **6.1 网络与通信故障**：
  - **现象**：连接中间件端口（`13001`）失败。
  - **排查命令**：
    1. 在运行端使用 `netstat -ano | findstr 13001` 检查是否有其他进程占用，或中间件程序是否死锁。
    2. 使用 `telnet 127.0.0.1 13001` 测试回路是否通畅。
- **6.2 组件与数据异常**：
  - **现象**：接收到的图片 Base64 字符串残缺，引发反序列化失败。
  - **恢复方案**：检查自定义代码块中 Socket 接收缓冲区参数，必须将其设为 `262144` 字节（256KB）以上，确保大报文不会在 TCP 传输中发生数据截断。

## 7. 运维服务单提交规范
- **7.1 日志获取路径与截取方法**：
  - RPA 引擎将日志记录在本地引擎的通用日志模块中，技术人员可通过检索本地日志输出，过滤 `error` 或 `Exception` 获取流程异常抛出时的堆栈详情。
- **7.2 故障提单要素**：
  - 包含发生异常的 `useraccount`、MySQL 数据库对应行的字段原始状态以及抛出错误时的日志文本。
- **7.3 磁盘与缓存清理**：
  - 本流程运行在后台且主要通过 Socket 传输 Base64 数据并输出文本日志，对日志文件的增长情况需进行定时巡检与清理。
