# 获取对应省信息.json 软屏蔽与无害化改造定位文档

> - **本篇目的**：物理留痕子流程 [获取对应省信息.json](file:///d:/data/Code/sendi/liucheng/登陆/省系统扫码登录服务端%20(2)/省系统扫码登录服务端/获取对应省信息.json) 中在不破坏任何 RPA 文件结构（0删除、0增减 Block）的前提下，实现旧代码分割逻辑的“无害化软屏蔽”，彻底避免 `list index out of range` 异常并保证流程能够被 RPA 平台安全导入。

---

## 1. 故障根源解析与“零结构破坏”设计

* **崩溃根源**：RPA 后台转换脚本会强行编译并运行被禁用的 Block（Index 8 和 Index 10）。在这两个 Block 试图读取不存在的 `字符串[1]` 时，由于省系统只剩下一个 Cookie，从而发生了 `list index out of range` 崩溃。
* **物理删除导致的新问题**：物理删除这些 Block 后，由于打破了 RPA 配置文件 JSON 内部的拓扑 ID 引用关系，RPA 控制台导入时会直接触发 `流程有破坏` 报错。
* **无害化改造策略**：**保留所有 Block 的 ID 和节点关系，仅修改触发越界报错的 inputs 参数，使其指向安全的数据，实现软屏蔽。**

## 2. 软屏蔽组件精确修改配置 (共 2 处)

本部分对 `获取对应省信息.json` 中仅有的 2 处引发数组越界的 inputs 节点执行内容调整，具体映射关系如下表：

### 2.1 修改一：Index 8 打印日志组件 (物理防崩)
* **组件 ID**：`PV2cezw4`
* **组件名称**：`Control.print`
* **原配置**：
  ```json
  "template_string": "字符串[1]"
  ```
* **替换为**（重定向为已存在的第一个元素）：
  ```json
  "template_string": "字符串[0]"
  ```

### 2.2 修改二：Index 10 变量赋值组件 (赋值防崩)
* **组件 ID**：`2FCHyGRO`
* **组件名称**：`Control.vardefine`
* **原配置**：
  ```json
  "init": "字符串[1]"
  ```
* **替换为**（直接赋空字符串字面量）：
  ```json
  "init": "''"
  ```

## 3. 校验与影响范围评估

* **结构零破坏**：由于未修改任何 Block ID，未删除任何节点，子流程与主流程的所有参数映射、变量列表与 SQL 语句均维持完全原样。
* **RPA 导入验证**：此修改不会破坏 JSON 树的任何引用和元数据，可确保流程能够 100% 成功导入到 RPA 开发控制台。
* **执行安全性**：后台生成的 Python 代码中，Index 8 将正常输出第一个 Cookie 值，Index 10 会给 `第二对参数` 安全地赋予空字串 `''`，绝不会越界。
